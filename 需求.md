补充信息（关于前端 Kelivo 的渲染能力）
- Kelivo 前端原生支持「深度思考」折叠块和工具调用展示。
- 界面上有独立的思考折叠区，也能把工具调用（如编辑记忆）单独显示在思考区，不混入正文。
- 因此功能2（思维链透传）和功能3（工具调用可见）大概率不用改前端：
  只要网关按 Anthropic 标准格式透传 thinking 块和 tool_use 块，Kelivo 会自动渲染。
- 实现要点：流式下把上游的 thinking / tool_use content block 按 Anthropic SSE 标准
  （content_block_start / content_block_delta / content_block_stop，type=thinking / tool_use）
  原样透传给前端即可。

调研结论（关于 thinking / budget，来自 Kelivo 源码 chat_api_service.dart）
- Kelivo 对 claude-opus/sonnet 4.6+ 走 adaptive 模式：thinking 字段固定发 {"type":"adaptive","display":"summarized"}。
- 前端调的思考预算不走 thinking.budget_tokens，而被换算成独立字段 output_config.effort（low/medium/high/xhigh/max）。
- 换算表：≤2000→low，≤20000→medium，≤32000→high，≤64000→xhigh，更大→max。
- 网关必须同时透传 thinking 和 output_config 两个字段，预算调节才生效（日志实测 max/low 生效）。
- adaptive 模式下模型自适应决定是否思考，同一档位有时出思考有时不出，属正常。

调研结论（关于 Kelivo 工具处理机制，来自 claude_official.dart）
- Kelivo 看到流里的 tool_use 块后，若自身配了工具（onToolCall 非空），会在 content_block_stop 时主动执行该工具，并把 assistant+tool_result 回传网关发起新一轮请求（会形成循环）。
- 但 recall/feel/update_profile 是网关注入并执行的，Kelivo 端没有实现，会导致错误卡片或循环。
- 因此功能3 采用「路线B」：网关继续内部执行工具（保持单一权威），用不会触发 Kelivo 执行引擎的方式做只读展示。先最小验证 Kelivo 渲染行为再定稿。
