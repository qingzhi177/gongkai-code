功能4：Token usage 透传
Kelivo 前端显示 token 消耗为 0。
原因：网关返回的响应里没有带 usage 字段。
请确保上游 API 返回的 usage 字段原样透传回前端。
流式模式下 usage 在最后一个 message_stop 事件里。
非流式模式下 usage 在响应 JSON 顶层。

功能5：上下文缓存优化
1. 缓存优化：确保 system prompt + L2画像 + 工具定义 作为请求的固定前缀，
   顺序稳定不变，利用 Anthropic prompt caching 命中。
   具体做法：把这三部分拼成 system 字段的内容，每次请求保持内容和顺序一致。
2. 滑动窗口：暂时不做。前端（Kelivo）自己控制发送多少条历史消息，
   网关原样转发，不截断不压缩。
3. 按需深检索：LLM 通过 recall 工具自主调取历史原文，
   网关不主动注入大段历史。
4. 暂不做摘要压缩。等遇到实际超长对话问题再考虑。

功能6：Dashboard 模型配置

2. 选择当前使用哪个供应商和模型
3. 修改后网关热重载（不重启服务）
4. 配置存 SQLite，API Key 加密存储，前端显示用 *** 脱敏
